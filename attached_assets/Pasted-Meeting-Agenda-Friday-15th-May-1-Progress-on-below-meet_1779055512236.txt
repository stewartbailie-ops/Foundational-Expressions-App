Security for client PII. Storing ID numbers, payslips, bank details, and policy documents puts us deep into POPIA "special personal information" territory. The minimum should be:
	• Encryption at rest for ID number, bank details, tax number — column-level AES via a key from a new PII_ENCRYPTION_KEY secret. Server decrypts on read.
	• Encrypted file storage for uploaded documents — files at rest are encrypted, only the advisor who owns the client can fetch.
	• Audit log — every read/write of sensitive PII gets a row in an audit table (who, what, when, IP). POPIA requires this.
	• Per-advisor isolation — backend queries hard-scope to clientRecord.advisorId === session.advisorId. No client document ever returnable to a different advisor.
	• Consent capture — when a lead is promoted to a client, capture explicit consent ("This advisor may store my personal documents"). Datestamp + IP.
	• Delete-on-request — POPIA right-to-erasure. We need a "permanently delete client" path that wipes encrypted data and audit-logs the deletion event itself.
	• One additional consideration: Stripe's South Africa onboarding requires a SA-registered business with a SA bank account linked — confirm those details are in order before the build session
